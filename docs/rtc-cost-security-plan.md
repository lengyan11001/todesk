# BHZN ToDesk RTC、成本与安全规划

更新时间：2026-06-12

## 目标

当前系统已经可以通过服务器中转实现远程观看和控制，但如果长期按纯中转架构承接免费用户，带宽成本会成为主要风险。本规划用于明确：

- 免费版能开放到什么程度，避免流量亏损。
- WebRTC/P2P 应该怎么改，降低服务器中转成本。
- TURN 节点如何做鉴权、限流和计费，避免被滥用。
- 当前系统还有哪些安全、成本、稳定性漏洞需要补。

## 当前现状

### 当前链路

```text
Agent -> WSS/Node 服务器 -> H5 控制端
```

特点：

- 画面、控制指令、文件传输都经过服务器。
- 服务器不做视频解码，只做 WebSocket 帧中转。
- 服务器带宽消耗会随观看端数量线性增长。
- 同一设备被多个 H5 同时观看时，服务器出站流量按观看端数量翻倍。

### 当前画质档位

Mac Agent 当前配置：

- 采集输出最长边：`1280`
- 常见 16:9 屏幕实际约：`1280x720`
- JPEG 质量：`0.54`
- 实测帧率：约 `1fps`

所以当前不是 ToDesk 免费版宣传的 `1080P/30fps`，更接近 `720P/低帧率/低码率`。

### 实测流量

设备：`4KIO-4ZVZ`  
计费假设：`0.5 元/GB`

| 场景 | 10 分钟服务器出站 | 折算每小时出站 | 出站成本 | 入站+出站成本 |
| --- | ---: | ---: | ---: | ---: |
| 桌面停留/轻操作 | 78.74 MB | 0.472 GB/h | 0.236 元/h | 0.47 元/h |
| 看电影 | 45.58 MB | 0.274 GB/h | 0.137 元/h | 0.27 元/h |

看电影反而更低，是因为当前 Agent 实际仍然只发约 `1fps`，不是视频级远控。

### 按 ToDesk 免费档顶格估算

ToDesk 免费档可参考 `1080P/30fps/4Mbps` 的级别。按 `4Mbps` 顶满计算：

- 单路出站：`1.8 GB/h`
- 只算出站：`0.9 元/h/观看端`
- 入站+出站都计费：`1.8 元/h/观看端`

这说明如果免费版开放到 `1080P/30fps/4Mbps`，必须有免费用户预算和强限流，否则长时间使用会直接烧钱。

## 成本模型

### 纯中转并发公式

```text
可承载并发控制端 ≈ 固定带宽 Mbps * 0.7 / 单路码率 Mbps
```

保留 30% 余量是为了避免 TCP/WebSocket 抖动、TLS、页面请求、心跳和突发帧导致整体卡顿。

### 固定带宽服务器估算

按单路 `4Mbps` 计算：

| 固定带宽 | 稳定并发 |
| ---: | ---: |
| 4 Mbps | 1 个 |
| 10 Mbps | 2 个 |
| 20 Mbps | 3-4 个 |
| 50 Mbps | 8-10 个 |
| 100 Mbps | 17-20 个 |

如果要支撑 `100` 个同时控制端，纯中转大约需要：

```text
100 * 4Mbps / 0.7 ≈ 572Mbps
```

也就是接近 `600Mbps` 固定带宽，成本会很高。

## ToDesk 盈利模型参考

ToDesk 的关键不是靠普通国内个人远控直接收费，而是免费增值：

- 免费版负责获客，限制画质、节点、功能和商用能力。
- 付费版卖更好的 RTC/VIP 节点、更高画质、更高帧率。
- 游戏版、设计版、性能版卖高码率、低延迟、专业场景。
- 插件卖全球节点、安卓免 Root、更多通道等高毛利功能。
- 团队/企业按用户、通道、设备管理、审计、安全策略、私有化收费。

我们应该采用类似分层，但要结合当前成本：

- 免费版必须限流、限时、限并发。
- 长时间、高画质、多观看端、团队使用必须收费。
- TURN/高速节点应作为会员或企业权益。

## 推荐产品分层

### 免费版

目标：获客和低频体验，不承接重度使用。

建议限制：

- 画质：`720P` 或 `1080P 低帧率`
- 帧率：`1-5fps`
- 并发：每个账号 `1` 个 active 控制端
- 时长：单次 `30-60` 分钟
- 每日免费额度：`30-60` 分钟
- 空闲断开：`3-5` 分钟无操作自动断开
- 文件传输：限大小、限速
- TURN 中转：免费版只给普通限速中转，或只在短时兜底使用

### 个人会员

目标：覆盖普通长时间使用成本。

建议权益：

- 更长时长或不限时长
- `1080P`
- `5-15fps`
- 更高码率上限
- 文件传输提速
- 设备数量增加
- 普通 TURN 兜底

定价建议：

- `19-29 元/月`
- `158-198 元/年`

### 高性能会员

目标：覆盖高流量场景。

建议权益：

- `2K/4K`
- `15-30fps`
- 更高码率
- 低延迟节点
- 高速 TURN 节点
- 多显示器、游戏键鼠、数位板等专业功能

定价建议：

- `68-148 元/月`
- `298-638 元/年`

### 团队/企业版

目标：按通道和管理能力收费。

建议权益：

- 团队设备池
- 成员权限
- 连接日志
- 文件传输审计
- 远程协助工单
- 通道并发数
- 专属节点
- 私有化部署

建议按通道收费，而不是无限放开账号：

- 团队通道：`99-199 元/月/通道`
- 企业通道：`800-3000 元/年/通道` 起

## WebRTC/P2P 改造方案

### 改造目标

从纯中转：

```text
Agent -> 服务器 -> H5
```

改为优先直连：

```text
Agent -> H5
```

服务器只负责：

- 登录鉴权
- 设备在线状态
- 绑定关系校验
- WebRTC 信令交换
- TURN 临时凭证签发
- P2P 失败后的兜底中转

### WebRTC 连接流程

1. H5 登录并选择设备。
2. 服务端确认用户已绑定该设备。
3. H5 创建 `RTCPeerConnection`。
4. H5 通过 WebSocket 发 `rtc-offer` 给服务端。
5. 服务端只转发给对应 Agent。
6. Agent 返回 `rtc-answer`。
7. 双方通过服务端交换 `ice-candidate`。
8. 双方优先尝试 P2P UDP 打洞。
9. P2P 失败则使用 TURN relay。
10. TURN 也失败则回退当前 WebSocket 中转。

### 传输通道

推荐：

- 视频：WebRTC video track
- 鼠标/键盘：WebRTC DataChannel
- 剪贴板：DataChannel
- 小文件控制消息：DataChannel
- 大文件：HTTP/对象存储/专门文件通道，不建议走视频通道

### Mac Agent 实现方向

为了达到最好效果，Mac 端不要继续 JPEG 截图流，建议改成：

- ScreenCaptureKit 采集屏幕
- VideoToolbox 硬编码 H.264
- WebRTC native SDK 发送视频 track
- DataChannel 接收鼠标、键盘、剪贴板、控制消息

这样可以获得：

- 更低码率
- 更低延迟
- 硬件编码
- 浏览器硬解码
- WebRTC 自带拥塞控制和丢包恢复

### 可承载能力变化

| 模式 | 服务器压力 | 当前 2 核 4G 机器粗略能力 |
| --- | --- | --- |
| 纯 WebSocket 中转 | 带宽线性增长 | 取决于固定带宽，20Mbps 约 3-4 个 4Mbps 控制端 |
| WebRTC P2P 成功 | 服务器只走信令 | 几百到上千在线/控制信令可承接 |
| TURN 中转 | 类似中转，吃带宽 | 按 TURN 固定带宽计算 |
| P2P + 部分 TURN | 只为失败用户烧带宽 | 取决于 TURN 失败比例 |

如果 P2P 成功率达到 `80%`，只有 `20%` 需要 TURN：

- `100` 个 active 控制端中，约 `20` 个烧 TURN 带宽。
- 按 `4Mbps` 计算，TURN 需要 `80Mbps`，实际建议 `100-150Mbps`。
- 比纯中转需要接近 `600Mbps` 明显更省。

## TURN/STUN 部署规划

### 安全组端口

正式公网 TURN 节点需要：

- `UDP 3478`：STUN/TURN
- `TCP 3478`：部分网络 UDP 不通时兜底
- `UDP 49152-65535`：TURN relay 端口段
- 可选 `TCP/UDP 5349`：TURN over TLS

来源正式环境需要 `0.0.0.0/0`，因为用户 IP 多变，不能靠白名单。

### 鉴权要求

TURN 不能匿名开放。必须使用：

- `use-auth-secret`
- 后端签发临时 username/password
- token 有效期 `30-60` 分钟
- 只给已登录、已授权、已绑定设备的用户签发
- 记录用户、设备、会话、TURN 流量
- 限制每个账号、设备、IP 的并发和带宽

### 不能做的事

- 不能开 anonymous TURN。
- 不能把固定账号密码写死在前端。
- 不能所有用户共用一个长期 TURN 密码。
- 不能只靠安全组来源 IP 做正式鉴权。

## 当前风险与漏洞

### 1. 设备身份认证偏弱

当前设备主要依赖 `deviceId + verificationCode`。这对商业化不够。

风险：

- 设备 ID 被猜测或撞库。
- 恶意客户端伪造设备上线。
- 老设备被替换或冒名。

改进：

- 首次绑定后生成 `deviceSecret`。
- Agent 本地安全保存 `deviceSecret`。
- 后续 WebSocket 连接使用 HMAC/JWT 签名。
- 服务端校验时间戳、nonce、防重放。
- 支持设备密钥轮换和解绑吊销。

### 2. 控制会话缺少端到端确认

当前服务端校验绑定关系后即可让控制端接入。

建议：

- 每次控制会话生成 `sessionId` 和短期 `sessionToken`。
- Agent 只接受服务端签名过的控制请求。
- 输入事件必须带 `sessionId`。
- Agent 端校验控制状态，不接受过期会话输入。

### 3. TURN 滥用风险

如果 TURN 开放不当，会被当代理滥用并烧流量。

必须补：

- 临时凭证。
- 按用户和设备限速。
- 按会话统计 TURN 用量。
- 超额降级或断开。
- 异常 IP/账号封禁。

### 4. 免费版成本失控

风险：

- 免费用户长时间挂着看屏幕。
- 多开浏览器观看同一设备。
- 高动态画面导致码率上升。
- 文件传输消耗额外带宽。

必须补：

- 每日免费时长。
- 单次连接时长。
- 空闲断开。
- 同账号并发限制。
- 同设备观看端限制。
- 免费版码率上限。

### 5. 文件传输内存风险

当前服务端文件上传使用 `express.raw`，请求体会进 Node 内存。虽然 nginx 当前 `client_max_body_size` 是 `20m`，但服务端配置是 `100MB`。

风险：

- 多个大文件并发上传导致 Node 内存上涨。
- 如果 nginx 放开到 100MB，风险更明显。

改进：

- 改成流式写盘。
- 文件上传走对象存储或专用上传服务。
- 加用户级、设备级并发限制。
- 加文件大小、每日流量、过期清理。

### 6. 服务器缺少实时带宽保护

当前已经补了按设备统计画面字节，但还没有全局带宽保护。

需要增加：

- 全局当前 Mbps 统计。
- 每设备 Mbps 统计。
- 每用户 Mbps 统计。
- 超限自动降画质/降帧率。
- 超限拒绝新免费会话。
- 管理后台展示实时带宽和会话。

### 7. 画面协议不是视频协议

当前 JPEG 帧通过 WebSocket 发送：

- 省实现成本。
- 但压缩效率低。
- 延迟和流畅度不如 WebRTC 视频。
- 高帧率时带宽会快速上升。

长期应迁移到 WebRTC 视频编码。

### 8. 权限和审计不足

商业化/企业版需要：

- 谁连接了哪台设备。
- 连接开始/结束时间。
- 控制时长。
- 文件发送记录。
- 输入事件摘要。
- 异常失败原因。
- 管理员审计导出。

## 更好的长期方案

### 方案 A：纯中转继续优化

优点：

- 实现快。
- 网络兼容性好。
- 用户体验可控。

缺点：

- 带宽成本高。
- 并发能力受固定带宽限制。

适合短期上线和小规模验证。

要补：

- 码率限制。
- 空闲断开。
- 带宽统计。
- 设备强认证。
- 文件传输流式化。

### 方案 B：WebRTC P2P + TURN 兜底

优点：

- 成本最低。
- 延迟更低。
- 画面更顺滑。
- 免费用户成本可控。

缺点：

- Agent 改造大。
- NAT/网络环境复杂。
- 必须维护 TURN 节点。

适合中长期主线。

### 方案 C：区域化中转节点

优点：

- 更容易控制质量。
- 比 P2P 更稳定。
- 可做 VIP 节点。

缺点：

- 成本仍然高。
- 需要节点调度。

适合会员、企业、跨运营商兜底。

### 推荐组合

```text
默认：WebRTC P2P
失败：普通 TURN/中转，免费限速
付费：VIP TURN/区域节点
企业：专属节点/私有化部署
```

## 落地优先级

### P0：商业化前必须补

- 设备 `deviceSecret` 强认证。
- 控制会话短期 token。
- 免费版并发、时长、空闲断开。
- 按用户/设备/会话统计流量。
- 服务端全局带宽保护。
- 文件传输流式写盘或限制到较小大小。

### P1：降低成本和提升体验

- 部署 coturn，启用临时凭证。
- 增加 WebRTC 信令消息。
- H5 增加 ICE 状态展示和失败回退。
- Mac Agent 做 WebRTC 原型。
- P2P 失败自动回退 WebSocket 中转。

### P2：会员和企业能力

- VIP 节点调度。
- 通道计费。
- 团队设备管理。
- 连接审计。
- 文件传输审计。
- 管理后台资源监控。

## 建议下一步

1. 先把免费版限制做进服务端，避免当前中转版本被长时间免费使用拖垮。
2. 补设备强认证，把 `deviceId + verificationCode` 升级为 `deviceId + deviceSecret + HMAC`。
3. 部署一台 coturn 测试节点，开放 UDP `3478` 和 relay 端口段。
4. H5 增加 ICE 探测页，先不传真实桌面，只验证 P2P/TURN 成功率。
5. Mac Agent 做 WebRTC 视频原型，再比较 `720P/1080P/5fps/15fps` 的真实流量。
6. 根据真实流量重新定免费额度和会员价格。
